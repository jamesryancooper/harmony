# ADR-0001: Adopt a Thin HTTP/MCP Wrapper for GitHub SpecKit (speckit)

**Status**: Accepted

## Context

Small teams need a consistent, auditable, spec-first workflow that produces predictable artifacts and enforces contracts in CI. Harmony’s Lean AI‑Accelerated Methodology prescribes Spec → ADR → BMAD, tiny PRs behind flags, contract tests, and SLOs with error budgets. Today, authors can run GitHub SpecKit locally, but agents/CI need a uniform, guard‑railed way to bootstrap, validate, and publish specs, with clear contracts and observability.

Constraints and drivers:
- Spec‑first with contracts under `packages/contracts/**` (OpenAPI + JSON‑Schema).  
- Harmony compliance: OWASP ASVS + NIST SSDF mapping; micro‑STRIDE in specs; CI gates (oasdiff, static/SCA, SBOM, secrets).
- Simplicity‑first: minimal new surface; prefer a thin wrapper over heavy service logic.
- Operability: feature‑flagged rollout, previews, instant rollback via promoting a prior deployment.

## Decision

Adopt a thin HTTP/MCP wrapper named “speckit” that standardizes GitHub SpecKit operations for agents and CI:
- Expose `init`, `validate`, `render`, and `diagram` under `/v1/speckit/*`, specified in `packages/contracts/openapi.yaml`.
- Validate Spec front matter and structure using JSON Schema (e.g., `packages/contracts/schemas/spec-frontmatter.schema.json`) and `specify check`.
- Keep contracts as the source of truth in `packages/contracts/**`; enforce OpenAPI breaking‑change diffs in CI.
- Treat ADRs and BMAD as PlanKit‑owned documents; this wrapper only orchestrates SpecKit and hands off to Dockit/DiagramKit for publishing/diagrams.
- Ship behind `flag.speckit`; internal audience first; add basic telemetry (run_id, idempotency_key) and budgets.

Scope: applies to the SpecKit wrapper surface (CLI/HTTP/MCP) and its contracts; does not replace Dockit/DiagramKit or PlanKit responsibilities.

## Consequences

- Pros:
  - Consistent, auditable spec runs; reproducible `.specify/` and `docs/specs/**` outputs.
  - Clear, testable boundaries via OpenAPI/JSON‑Schema; CI can block drift (oasdiff, schema validation).
  - Aligns with Harmony’s spec‑first, tiny‑PR, flag‑gated delivery and compliance mapping (ASVS/SSDF, STRIDE).
  - Minimal surface area; delegates rendering and deep publishing concerns to Dockit/DiagramKit.
- Cons:
  - Additional wrapper to maintain in lock‑step with upstream SpecKit UX changes.
  - Another operational surface (if HTTP) to monitor and budget; potential duplication of CLI behavior.
- Follow‑ups:
  - Implement endpoints and stub handlers; add contract tests and negative cases from STRIDE.
  - Wire CI gates: `oasdiff` for OpenAPI, schema validation for front matter, preview smoke.
  - Add feature flag plumbing and observability (structured logs, traces with run_id).
- Reversal trigger(s):
  - Wrapper complexity or operational cost outweighs benefits for the 2‑dev team.
  - Upstream SpecKit adds an equivalent, stable, agent‑friendly API that renders the wrapper redundant.
  - Persistent SLO violations or security concerns not mitigated by the design.
